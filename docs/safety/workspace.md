# Workspace safety

How AutoSkillit isolates pipeline work from your real source tree, what
`merge_worktree` actually does on a green run, and how the editable-install
guard prevents broken CLI installations after cleanup.

## Clone-based isolation

`workspace/clone.py:clone_repo` creates a fresh clone of the source repository
into `autoskillit-runs/<run>-<timestamp>/` and rewrites the clone's `origin`
remote so the orchestrator can never accidentally push to the user's working
tree. The orchestrator runs every subsequent step against the clone, never
the source.

## Origin re-pointing

After the clone, `clone_repo` rewrites `origin` to the canonical GitHub
remote (resolved by `execution/remote_resolver.resolve_remote_repo`,
preferring `upstream` over `origin` and de-duplicating clone-local remotes).
This guarantees that even if the orchestrator runs `git push origin`, it
hits the upstream repository rather than the user's local clone path.

## Clone registry

`workspace/clone_registry.py` maintains a shared file-based registry of
clones eligible for deferred batch cleanup. The registry is read by
`batch_cleanup_clones` to delete stale clones in bulk after a recipe finishes.
The registry uses atomic JSON writes so concurrent recipe runs cannot corrupt
each other.

## `merge_worktree` 13-step pipeline

`server/git.py:perform_merge` runs every step in order; failure at any step
returns a `MergeFailedStep` value to the orchestrator and aborts the merge:

1. **path validation** — the worktree path must exist and be a real git
   worktree, not a regular directory.
2. **protected branch check** — `core/branch_guard.is_protected_branch`
   denies merges into branches in `safety.protected_branches`.
3. **branch detection** — resolve the worktree's HEAD branch and the merge
   target branch.
4. **dirty tree check** — both the worktree and the target must be clean.
5. **pre-merge test gate** — run the project's test command via `test_check`;
   abort on failure.
6. **fetch** — `git fetch origin` to refresh remote refs.
7. **pre-rebase check** — abort if the worktree is already in a half-merged
   state.
8. **merge-commits-detected stop** — fail loudly if the worktree contains
   merge commits; AutoSkillit only supports linear history.
9. **rebase** — rebase the worktree onto the latest target.
10. **generated-file cleanup** — remove autogenerated files (`hooks.json`,
    `settings.json`) before the second test gate.
11. **post-rebase test gate** — re-run `test_check` after the rebase to
    catch latent conflicts.
12. **merge** — fast-forward merge into the target branch.
13. **editable install guard** — see below.

## Editable install guard

`server/_editable_guard.py:scan_editable_installs_for_worktree` runs after
the merge but before any cleanup. It scans the user's Python tools for
editable installs that point inside the worktree directory. If any are
found, the merge result is `MERGE_SUCCEEDED_CLEANUP_BLOCKED` and the
worktree is left intact so the user can re-install from the source path
before deletion. Without this guard, deleting a worktree that hosted an
editable install would silently break the user's `autoskillit` CLI.

## Workspace markers

`autoskillit workspace init` creates a `.autoskillit-workspace` marker file
inside a directory, marking it as a sanctioned scratch directory for tests
and reset operations. `reset_workspace` and `reset_test_dir` refuse to act on
any directory without this marker — preventing accidental wipes of the wrong
filesystem path. The marker name is configurable via
`safety.reset_guard_marker`.

## Retry-worktree for context exhaustion

When a long `implement-worktree-no-merge` session hits its context limit, the
worker session terminates with `needs_retry=true` and the worktree is left
intact on disk with all commits made up to that point. The orchestrator MUST
route to `/autoskillit:retry-worktree` against the same worktree path
(returned in `result.worktree_path`) — never re-call
`/autoskillit:implement-worktree-no-merge`, which would create a brand-new
timestamped worktree and discard the partial progress.

## Isolation rules in `recipe/rules_isolation.py`

Two semantic rules enforce the isolation invariants at recipe-validation
time:

- `source-isolation-violation` — fires when a recipe step targets the source
  directory instead of the clone.
- `git-mutation-on-source` — fires when a recipe runs a git mutation
  command (`git commit`, `git push`, `git rebase`) against the source remote.

Both rules cause `validate_recipe` to fail before the orchestrator can ever
run the recipe.
